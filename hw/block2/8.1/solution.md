### Отсутствие wait, lock, deadlock, starvation freedom
Пусть нам на 4d20 всегда выпадает 4, а планировщик распределяет кванты планирования так, что верно следующее
(будем отмечать исполнение F.N потоком X как X.N)  
A.4 -> B.4 -> {оба флага подняты} -> A.5 -> B.5 -> A.8 -> B.8 -> A.9 -> B.9 -> повторить  
В этом примере ни один вызов `lock()` никогда не вернется, 
а следовательно метод не является ни wait-, ни lock, ни deadlock-free (два потока ждут друг друга не выполняя работы)  
Раз starvation-freedom -> deadlock-freedom, то !deadlock-freedom -> !starvation-freedom.
Таким образом, `foo` также не гарантирует starvation-freedom.

### Obstruction-freedom
Заметим, что F.3 - F.9 - это цикл, притом единственный. 
Пусть потоки A, B работали некоторое время, после чего поток B был "заморожен".
В каком бы состоянии не находился поток A, он либо исполнит A.7, и выйдет из `lock()`, 
либо дойдет до A.3 путями:
- A.4 -> A.5 -> A.3
- A.4 -> A.5 -> A.6 -> A.3
- A.4 -> A.5 -> A.6 -> A.7
- A.8 -> A.9 -> A.3,

или их концевыми подпутями, так что рассмотрим поток A с момента A.3  
A.3 -> A.4 -> {flags == (1, ?)} -> A.5

при flags == (1, 0):  
A.5 -> A.6 -> {состояние флагов не изменялось} -> A.7

при flags == (1, 1):  
A.5 -> A.8 -> A.9 -> {flags == (0, 0)} -> A.3 -> A.4 -> {flags == (1, 0)} -> {см. предыдущий случай}

При "заморозке" потока B поток А завершается не более чем за 3*7 (длина цикла) "операций" (строк, каждая из которых - конечное число операций)=> 
`foo` гарантирует obstruction-freedom
